# Provenance Guard — Planning


## The problem in one paragraph

Platforms where people share original creative work face a new question: was a submitted
piece written by a human, or generated by an AI and passed off as human? **Provenance Guard**
is a backend any such platform can plug into. It accepts a piece of text, runs it through a
**multi-signal detection pipeline**, produces a **calibrated confidence score** (not a binary
verdict), surfaces a **plain-language transparency label** to readers, lets misclassified
creators **appeal**, and records every decision in a **structured audit log** behind a
**rate-limited** submission endpoint. The goal is not to police creativity — it is to protect
attribution and give readers honest context, while *acknowledging uncertainty* rather than
pretending detection is solved.

## Architecture

### The path a submission takes (architecture narrative)

A single piece of text enters at `POST /submit` with a `text` body and a `creator_id`. The
request first passes the **rate limiter** (Flask-Limiter). If it is under the limit, the text
is fanned out to **three independent detection signals**, each returning a probability that the
text is AI-generated, `P(AI) ∈ [0,1]`:

1. **LLM holistic judgment** (Groq `llama-3.3-70b-versatile`) — semantic/stylistic read.
2. **Stylometric heuristics** (pure Python) — structural statistics of the text.
3. **AI-tell phrase density** (pure Python) — lexical fingerprints of LLM phrasing.

The **scoring** component combines the three signals with documented weights into a single
`combined_score`, derives a **confidence** value (decisiveness) and an **attribution**
(`likely_ai` / `uncertain` / `likely_human`), and the **labels** component turns that into the
exact transparency text a reader sees. A unique `content_id` is minted, a **content record**
is written to the content store (status `classified`), an entry is appended to the **audit
log**, and the response is returned to the caller.

An appeal enters at `POST /appeal` with the `content_id` and the creator's `creator_reasoning`.
The content record's status is updated to `under_review`, an `appealed` event is appended to the
audit log alongside the original decision, and a confirmation is returned. There is **no**
automated re-classification — appeals are queued for a human reviewer.

`GET /log` returns the most recent audit entries as JSON so the decision history is inspectable.

## The five required design questions

### 1. Detection signals — what are your 2+ signals, what does each measure, what does each output, and how are they combined?

Three signals (the third makes this an **ensemble**, the stretch feature). Each is normalized
to return **`P(AI)` ∈ [0,1]** so they can be combined cleanly.

| Signal | What it measures | Output | Why it differs |
|---|---|---|---|
| **1. LLM holistic** (Groq `llama-3.3-70b-versatile`) | Semantic & stylistic coherence — does this *read* like AI? | JSON `{ai_probability: 0–1, reasoning}`, used directly as `P(AI)` | Holistic, meaning-aware judgment |
| **2. Stylometric heuristics** (pure Python) | Structural uniformity: sentence-length variance (burstiness), type-token ratio, punctuation density, mean sentence length | Rubric → score 0–1 | Pure statistics — no meaning, no model |
| **3. AI-tell phrase density** (pure Python) | Frequency of formulaic LLM phrasing ("Furthermore", "It is important to note", "plays a crucial role", "In conclusion") | Density → score 0–1 | Lexical fingerprints, orthogonal to both above |

These are genuinely **distinct**: one is semantic (a model's read), one is structural
(statistics of sentence shape), one is lexical (specific phrasings). They fail in different
ways, so combining them is more informative than any one alone.

**Combination:** weighted average, `combined_score = 0.5·llm + 0.3·stylometric + 0.2·phrase`.
The LLM gets the most weight (best signal in practice), stylometrics next (robust but coarse),
phrase density least (easiest to evade, so a supporting vote rather than a deciding one). The
weights live in `detection/scoring.py` as named constants and are documented in the README.

### 2. Uncertainty representation — what does 0.6 mean, how do raw outputs map to a calibrated score, and what thresholds separate the three classes?

`combined_score` is `P(AI)`. From it we derive a **confidence** value that is *decisiveness*:

```
confidence = 2 · |combined_score − 0.5|        # ∈ [0,1]
```

- `combined_score = 0.5` → `confidence = 0.0` → the system genuinely cannot tell → **uncertain**.
- `combined_score = 0.95` → `confidence = 0.90` → very decisive.
- `combined_score = 0.6` → `confidence = 0.20` → leans AI but **weakly** — this lands in the
  uncertain band, which is exactly the honest answer for a 0.6.

This directly answers the hint "decide what 0.5 means to a user first": **0.5 means maximum
uncertainty**, and confidence grows as the score moves toward either pole.

**Attribution thresholds (asymmetric on purpose):**

| Condition on `combined_score` | Attribution |
|---|---|
| `≥ 0.70` | `likely_ai` |
| `≤ 0.40` | `likely_human` |
| otherwise (0.40 – 0.70) | `uncertain` |

The bar to call something AI (0.70) is **higher** than the bar to call it human (0.40). On a
writing platform a **false positive — labeling a human's work as AI — is worse** than a false
negative, so the system is deliberately slower to accuse. A 0.51 lands in `uncertain`; a 0.95
lands in `likely_ai` with high confidence — meaningfully different labels, as required.

### 3. Transparency label design — exact text for all three variants

Each label is a single plain-language string. `{pct}` = `round(confidence·100)`.

- **High-confidence AI (`likely_ai`):**
  `🤖 Likely AI-generated — our analysis strongly indicates this text was produced by an AI system. Confidence: {pct}%. The creator can appeal this result.`

- **High-confidence human (`likely_human`):**
  `✍️ Likely human-written — our analysis strongly indicates a person wrote this text. Confidence: {pct}%.`

- **Uncertain (`uncertain`):**
  `❓ Uncertain origin — we could not determine with confidence whether this text was written by a person or an AI. Treat attribution as inconclusive. Confidence: {pct}%.`

Design choices: the AI label *invites the appeal* (since that is the harmful-error case); the
uncertain label explicitly tells readers to **treat attribution as inconclusive** rather than
implying a hidden verdict; all three lead with an emoji + a short verdict so a non-technical
reader gets the gist in one glance, with the confidence percentage as supporting detail.

### 4. Appeals workflow — who appeals, what they provide, what the system does, what a reviewer sees

- **Who:** the content's creator, when they believe they were misclassified (most importantly,
  a human wrongly labeled AI).
- **What they provide:** the `content_id` from the original `/submit` response and a free-text
  `creator_reasoning` explaining why the classification is wrong.
- **What the system does:** looks up the content record; sets its status to **`under_review`**;
  appends an `appealed` event to the audit log that sits *alongside* the original decision and
  stores the `appeal_reasoning`; returns a confirmation. **No automated re-classification** —
  the original score/label are preserved as the record of what the system decided.
- **What a reviewer sees in the queue:** via `GET /log`, every `under_review` item with the
  original attribution/confidence/signal scores **and** the creator's reasoning, so a human has
  the full context to overturn or uphold the decision.

### 5. Anticipated edge cases — at least two specific scenarios the system handles poorly

- **A short, repetitive poem with simple vocabulary** (e.g. a child's rhyme): low type-token
  ratio and uniform short sentences look "AI-like" to the stylometric signal, risking a
  **false positive** on genuine human creative work. Mitigation: the asymmetric 0.70 AI
  threshold and the LLM signal (which reads it as a poem) usually pull it back to `uncertain`.
- **Formal human academic / policy writing** (e.g. an economics paragraph): naturally uniform
  sentence structure *and* heavy use of transition phrases ("Furthermore", "the relationship
  between") trip both the stylometric and phrase signals — the system's biggest false-positive
  risk. Documented as a known limitation; the LLM weight (0.5) is the main counterbalance.
- **Lightly-edited AI text** (AI draft a human polished): genuinely ambiguous provenance. The
  design *wants* this in the `uncertain` band — honest, rather than falsely confident.
- **Very short input** (one line / a haiku): too little signal for any method; biases toward
  `uncertain` because no signal reaches a confident pole.

## AI Tool Plan

For each implementation milestone, what spec context to provide, what to ask the AI to
generate, and how to verify before wiring it in.

### Milestone 3 — submission endpoint + first signal
- **Provide:** the *Detection signals* section (Signal 1) + the *Architecture* diagram +
  the `/submit` request/response contract.
- **Ask for:** a minimal Flask app skeleton, the `POST /submit` route stub (returning a
  hardcoded response first), and the **LLM signal function** (`llm_signal(text) -> P(AI)`)
  calling Groq `llama-3.3-70b-versatile` with a prompt that returns structured JSON.
- **Verify:** call `llm_signal()` directly on a few obvious AI and obvious human inputs and
  inspect the scores *before* wiring it into the route; then `curl /submit` and confirm the
  JSON shape (`content_id`, `attribution`, `confidence`, `label`) and that an audit entry is
  written. Check the route structure matches the API contract; edit, don't paste blindly.

### Milestone 4 — second signal + confidence scoring
- **Provide:** the *Detection signals* (Signals 2 & 3) + *Uncertainty representation* sections
  (weights + thresholds) + the diagram.
- **Ask for:** the **stylometric** function and the **AI-tell phrase** function as standalone,
  testable units, plus the **scoring** function that combines all signals into `combined_score`,
  `confidence`, and `attribution` exactly per the documented weights/thresholds.
- **Verify:** run each signal on the four spec test inputs (clear-AI, clear-human, two
  borderline). Confirm the combined scores **vary meaningfully** and that the AI tool's scoring
  matches my thresholds (AI tools sometimes invent reasonable-looking ranges that silently
  diverge — correct it if so). Confirm all three label classes are reachable.

### Milestone 5 — production layer (label, appeals, rate limit, audit)
- **Provide:** the *Transparency label design* (3 variants) + *Appeals workflow* sections + the
  diagram + the audit-log schema.
- **Ask for:** the **label generation** function (score → exact variant text), the
  `POST /appeal` endpoint (status update + audit append), the **Flask-Limiter** config on
  `/submit`, and the structured **audit log** writer/reader (JSON).
- **Verify:** submit inputs producing each label variant and confirm the text matches this spec
  verbatim; appeal a real `content_id` and confirm status flips to `under_review` and is logged;
  fire 12 rapid requests and confirm the first 10 return `200` and the rest `429`; `GET /log`
  shows ≥3 structured entries including an appeal.
